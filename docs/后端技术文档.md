## 2. 系统架构

```
客户端 (移动APP)
    │
    │  所有业务请求携带 HTTP Header:
    │    X-API-Key: <api-key>
    │    Content-Type: application/json (JSON请求)
    │    Content-Type: multipart/form-data (上传视频)
    │
    ▼
┌──────────────────────────────────────┐
│  ApiKeyFilter                         │
│  - /actuator/** → 无需认证直接放行     │
│  - Root Key → 全部权限，不限流        │
│  - 设备 Key → 查 Redis/MySQL，权限分级│
│  - Redis 故障 → 自动降级查 MySQL      │
│  - 校验失败 → 401 JSON               │
│  - 无状态会话，CSRF 已关闭            │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  RateLimitFilter                      │
│  - Root Key → 不限流                 │
│  - 设备 Key → 按配额限制（默认20/分） │
│  - Redis 故障 → 安全放行不限流        │
│  - 超限 → 429 JSON                   │
└──────────────────────────────────────┘
    │
    ▼
Spring Boot Controller 层
    │
    ├─ POST /api/v1/detect       → 视频上传 + 反诈检测
    ├─ GET  /api/v1/history      → 检测历史查询（游标/分页两种模式）
    ├─ GET|POST|DELETE /api/v1/blacklist/{authority|global|temp} → 黑名单 CRUD
    ├─ GET|POST|PUT|DELETE /api/v1/keys/... → API Key 管理 + 设备注册
    │
    ▼
Service 层
    │
    ├─► Redis  ← 三道黑名单检查（authority → global → temp）
    │             Redis 故障 → 安全放行不检查黑名单
    ├─► Python AI 服务 (POST /v1/analyze/video) ← HTTP multipart 调用
    │             超时/5xx → 指数退避重试（最多 3 次）
    │             4xx → 立即失败不重试
    └─► MySQL (t_detection_record, t_api_key) ← MyBatis 持久化
```

## 3. 认证机制 — 动态 API Key 体系

系统支持两类 API Key：

| 类型 | 说明 | 存储位置 | 生命周期 |
|------|------|---------|---------|
| **Root Key** | 管理员引导密钥，拥有全部权限 | `application.yml` (`api.key`) | 永久，除非重新部署 |
| **设备 Key** | 动态生成，绑定设备，权限可配 | MySQL `t_api_key` + Redis 缓存 | 可吊销、可设过期时间 |

### 3.1 请求格式

```
GET|POST|DELETE /api/v1/xxx
Header: X-API-Key: <api-key>
```

### 3.2 无需认证的路径

| 路径 | 说明 |
|---|---|
| `/actuator/health` | AI 服务健康检查 |
| `/actuator/info` | 应用基本信息 |

### 3.3 认证流程

```
1. 请求到达 → ApiKeyFilter
2. 路径是 /actuator/** → 直接放行
3. 提取 X-API-Key 头
4. 匹配 Root Key → 授予全部权限（ROLE_DETECT + ROLE_HISTORY + ROLE_ADMIN），不限流
5. 不匹配 Root Key → 查 Redis 缓存 apikey:{keyValue}
   ├─ 命中 → 检查 status=ACTIVE、未过期 → 授予对应权限
   └─ 未命中 → 查 MySQL t_api_key
       ├─ 找到且有效 → 写 Redis 缓存（TTL 5min），授予权限
       └─ 找不到 / 已吊销 / 已过期 → 401
6. 认证通过 → RateLimitFilter 按 Key 配额限流
```

### 3.4 权限体系

| 权限标识 | Spring Security Role | 可访问的接口 |
|---------|---------------------|------------|
| `DETECT` | `ROLE_DETECT` | `POST /api/v1/detect` |
| `HISTORY` | `ROLE_HISTORY` | `GET /api/v1/history` |
| `ADMIN` | `ROLE_ADMIN` | `/api/v1/blacklist/**`、`/api/v1/keys/**` |

- Root Key 拥有全部三种权限
- 设备 Key 默认 `DETECT,HISTORY`（仅可调用上传检测 + 历史查询）

### 3.5 认证失败响应

```json
// HTTP 401
{
  "code": 401,
  "message": "Missing or invalid API key"
}
```

### 3.6 配置方式

- Root Key 通过 `api.key` 配置（环境变量 `API_KEY`，dev 环境默认 `dev-api-key-123`）
- 设备 Key 通过 API 动态管理（见第 5 节）

## 4. 统一响应格式

所有接口返回统一的 `ApiResponse<T>` 结构：

```json
// 成功 (200)
{
  "code": 200,
  "message": "success",
  "data": { ... }     // 具体业务数据，可能为 null
}

// 失败 (4xx / 5xx)
{
  "code": 400,
  "message": "错误描述",
  "data": null        // 字段会被省略（@JsonInclude NON_NULL）
}
```

- `data` 为 null 时 JSON 中自动省略该字段
- 成功时 `code` 始终为 200
- 业务错误和系统错误用不同 code 区分

## 5. API 端点详解

### 5.1 视频检测 `POST /api/v1/detect`

**核心接口。上传短视频进行反诈风险检测。**

#### 请求 (multipart/form-data)

```
POST /api/v1/detect
Header: X-API-Key: <api-key>
Content-Type: multipart/form-data

参数（都是必填）:
  deviceId   String    设备唯一标识（如 IDFA / OAID）
  authorId   String    视频作者 ID（抖音号/平台标识）
  video      File      视频文件（≤ 10MB）
```

#### 响应 — 正常检测结果

```json
// 高风险 (HIGH)
{
  "code": 200,
  "message": "success",
  "data": {
    "riskLevel": "HIGH",
    "score": 0.78,
    "reason": "综合评分过高",
    "transcription": "视频语音识别文本..."
  }
}

// 中风险 (MEDIUM)
{
  "code": 200,
  "message": "success",
  "data": {
    "riskLevel": "MEDIUM",
    "score": 0.44,
    "reason": "综合评分中等",
    "transcription": "..."
  }
}

// 安全 (SAFE)
{
  "code": 200,
  "message": "success",
  "data": {
    "riskLevel": "SAFE",
    "score": 0.17,
    "transcription": "..."
  }
}
```

**SAFE 时 `reason` 字段为 null，JSON 中省略。**

#### 响应 — 黑名单命中（不调用 AI，不写数据库）

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "riskLevel": "HIGH",
    "source": "authority",
    "reason": "权威黑名单发布者"
  }
}
```

**黑名单命中时字段说明：**

| 字段 | 说明 |
|---|---|
| `riskLevel` | 固定 `"HIGH"` |
| `source` | `"authority"` / `"global"` / `"temp"` |
| `reason` | 来自黑名单记录的原因描述 |
| `score` | **不存在**（没有 AI 评分，JSON 省略此字段） |
| `transcription` | **不存在**（没有 AI 调用，JSON 省略此字段） |

> 提示：客户端可根据 `source` 字段是否存在来判断是黑名单命中还是正常检测结果。

#### 检测流程（后端内部逻辑）

```
1. 检查黑名单 (authority → global → temp)
   └─ 命中 → 立即返回 HIGH，不调 AI，不写库

2. 调用 AI 服务
   POST {AI_SERVICE_URL}/v1/analyze/video
   发送视频文件 (multipart)
   - 连接超时 5s，读取超时 30s
   - 超时 / 5xx → 指数退避重试（最多 3 次，间隔 1s / 2s / 4s）
   - 4xx（客户端错误）→ 立即失败，不浪费重试
   - 重试耗尽 → 抛出 AiServiceException，返回 502
   期望返回 JSON:
   {
     "aiGlitchProb": 0.7,     // AI 生成/伪造概率 (0-1)
     "violenceProb": 0.3,     // 暴力内容概率 (0-1)
     "transcription": "...",  // 视频语音识别文本
     "keywordHit": false      // 是否命中反诈敏感词
   }
   └─ 调用失败 → 抛 RuntimeException，返回 500

3. 风险评分计算
   - keywordHit == true → HIGH (score=1.0)
   - 否则: score = 0.7 × aiGlitchProb + 0.3 × violenceProb
            score >= 0.6 → HIGH
            score >= 0.3 → MEDIUM
            score <  0.3 → SAFE

4. 写入 MySQL 检测记录

5. 若结果为 HIGH → 自动加入临时黑名单 (24h TTL)

6. 返回检测结果
```

### 5.2 检测历史 `GET /api/v1/history`

查询某设备的历史检测记录。支持两种分页模式。

#### 模式 A：偏移分页（浏览模式）

```
GET /api/v1/history?deviceId=d1&page=1&size=20
Header: X-API-Key: <api-key>
```

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---|---|---|
| `deviceId` | String | ✅ | — | 设备标识 |
| `page` | int | ❌ | 1 | 页码（从 1 开始） |
| `size` | int | ❌ | 20 | 每页条数（1-100，超出自动截断为默认值） |

#### 模式 B：游标分页（增量同步）

```
GET /api/v1/history?deviceId=d1&afterId=200&size=20
Header: X-API-Key: <api-key>
```

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `deviceId` | String | ✅ | 设备标识 |
| `afterId` | Long | ✅ | 上次同步到的最新记录 ID，返回 id > afterId 的记录 |
| `size` | int | ❌ | 本次拉取条数（1-100，默认 20） |

> 两种模式互斥：传了 `afterId` 则忽略 `page`，走游标模式。

#### 响应

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [
      {
        "id": 301,
        "deviceId": "d1",
        "authorId": "author123",
        "riskLevel": "HIGH",
        "score": 0.85,
        "createdAt": "2026-05-28T15:30:00"
      },
      {
        "id": 300,
        "deviceId": "d1",
        "authorId": "author456",
        "riskLevel": "SAFE",
        "score": 0.15,
        "createdAt": "2026-05-28T14:20:00"
      }
    ],
    "total": 150,
    "page": 1,
    "size": 20,
    "hasMore": true,
    "latestId": 301
  }
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `records` | Array | 检测记录列表，按 id 倒序（最新在前） |
| `records[].id` | Long | 记录主键，自增，**唯一且单调递增**，可作为同步游标 |
| `records[].deviceId` | String | 设备 ID |
| `records[].authorId` | String | 作者 ID |
| `records[].riskLevel` | Enum | `HIGH` / `MEDIUM` / `SAFE` |
| `records[].score` | Double | 风险评分 (0.0-1.0) |
| `records[].createdAt` | DateTime | 检测时间 (ISO 8601) |
| `total` | Long | 该设备总记录数 |
| `page` | int | 当前页码（游标模式固定为 1） |
| `size` | int | 实际每页条数 |
| `hasMore` | boolean | 是否还有更多数据（客户端翻页/继续同步的依据） |
| `latestId` | Long | 本次返回记录中的最大 ID，用于下次增量同步的 `afterId` 入参 |

**增量同步推荐策略**：App 启动时传上次存储的 `latestId` 作为 `afterId` 获取增量，无新数据时 `records` 为空数组。

### 5.3 黑名单管理

黑名单分三种类型，优先级从高到低：

| 类型 | URL 路径 | Redis 结构 | 生命周期 | 来源 |
|---|---|---|---|---|
| 权威黑名单 | `authority` | Set | 永久 | 官方平台推送 |
| 全局黑名单 | `global` | Set | 永久 | 管理员手动添加 |
| 临时黑名单 | `temp` | String (带 TTL) | 24 小时 | 检测到 HIGH 时自动添加 |

#### 5.3.1 列出黑名单

```
GET /api/v1/blacklist/authority
GET /api/v1/blacklist/global
GET /api/v1/blacklist/temp
Header: X-API-Key: <api-key>
```

响应示例：
```json
{
  "code": 200,
  "message": "success",
  "data": ["author123", "author456"]
}
```

#### 5.3.2 检查单个作者

```
GET /api/v1/blacklist/authority/{authorId}
GET /api/v1/blacklist/global/{authorId}
GET /api/v1/blacklist/temp/{authorId}
GET /api/v1/blacklist/check/{authorId}    ← 检查所有三种黑名单
Header: X-API-Key: <api-key>
```

响应示例：

```json
// authority / global 的返回
{
  "code": 200,
  "message": "success",
  "data": { "blacklisted": true }
}

// temp 的返回（多一个剩余时间）
{
  "code": 200,
  "message": "success",
  "data": {
    "blacklisted": true,
    "ttlSeconds": 43200
  }
}

// check 的返回（汇总三种）
{
  "code": 200,
  "message": "success",
  "data": {
    "hit": true,
    "source": "temp",
    "reason": "临时黑名单发布者: 关键词命中"
  }
}
```

#### 5.3.3 添加黑名单

```
POST /api/v1/blacklist/authority
POST /api/v1/blacklist/global
POST /api/v1/blacklist/temp
Header: X-API-Key: <api-key>
Content-Type: application/json

Body (authority / global):
{ "authorId": "author123" }

Body (temp):
{ "authorId": "author123", "reason": "手动添加原因" }
```

#### 5.3.4 移除黑名单

```
DELETE /api/v1/blacklist/authority/{authorId}
DELETE /api/v1/blacklist/global/{authorId}
DELETE /api/v1/blacklist/temp/{authorId}
Header: X-API-Key: <api-key>
```

### 5.4 API Key 管理

**所有 Key 管理接口需要 `ROLE_ADMIN` 权限（Root Key 或带有 ADMIN 权限的设备 Key）。**

#### 5.4.1 设备注册 `POST /api/v1/keys/register`

设备首次启动时调用，获取专属 API Key。使用 Root Key 认证。

```
POST /api/v1/keys/register
Header: X-API-Key: <root-key>
Content-Type: application/json

请求体:
{
  "deviceId": "device-001",           // 必填，设备唯一标识
  "deviceName": "张三的iPhone"         // 可选，管理后台辨识用
}

响应 200:
{
  "code": 200,
  "message": "success",
  "data": {
    "apiKey": "a1b2c3d4e5f67890abcdef1234567890",
    "expiresAt": null                   // null = 永不过期
  }
}
```

- 同一 `deviceId` 重复注册 → 返回已有 Key（不新建）
- 生成算法：`UUID.randomUUID().toString().replace("-", "")`（32 位十六进制）
- **Key 只在注册/创建时返回一次，客户端需安全存储**

#### 5.4.2 列出所有 Key `GET /api/v1/keys`

```
GET /api/v1/keys
Header: X-API-Key: <ADMIN key>

响应 200:
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "keyName": "张三的iPhone",
      "deviceId": "device-001",
      "permissions": "DETECT,HISTORY",
      "status": "ACTIVE",
      "rateLimit": 20,
      "lastUsedAt": "2026-06-05T10:30:00",
      "expiredAt": null,
      "createdAt": "2026-06-05T09:00:00"
    }
  ]
}
```

注意：**列表接口不返回 `keyValue`**，保护密钥安全。

#### 5.4.3 查看单个 Key `GET /api/v1/keys/{id}`

返回完整 Key 信息（包括 `keyValue`）。

#### 5.4.4 手动创建 Key `POST /api/v1/keys`

```
POST /api/v1/keys
Header: X-API-Key: <ADMIN key>
Content-Type: application/json

请求体:
{
  "keyName": "测试Key",                // 必填
  "permissions": "DETECT,HISTORY",     // 可选，默认 DETECT,HISTORY
  "rateLimit": 30,                     // 可选，默认 20
  "expiredAt": "2026-12-31T23:59:59"   // 可选，过期时间（null = 永不过期）
}

响应 200:
{
  "code": 200,
  "message": "success",
  "data": {
    "apiKey": "新生成的Key...",
    "expiresAt": "2026-12-31T23:59:59"
  }
}
```

#### 5.4.5 修改 Key `PUT /api/v1/keys/{id}`

可修改字段：`keyName`、`permissions`、`rateLimit`、`expiredAt`（传 null 清除过期时间）。
修改后立即清除 Redis 缓存。

#### 5.4.6 吊销 Key `POST /api/v1/keys/{id}/revoke`

- 设置 `status = REVOKED`
- 清除 Redis 缓存
- 吊销后该 Key 立即失效

```json
{ "code": 200, "message": "success", "data": null }
```

#### 5.4.7 删除 Key `DELETE /api/v1/keys/{id}`

从 MySQL 和 Redis 中彻底删除。

### 5.5 限流机制

每个设备 Key 有独立的 `rate_limit` 配额（默认 20 次/分钟）。Root Key 不限流。

#### 限流响应

```
HTTP 429 Too Many Requests
Retry-After: 45
X-RateLimit-Limit: 20
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1717545650

Body:
{
  "code": 429,
  "message": "请求过于频繁，请稍后再试"
}
```

#### 响应头说明

| 响应头 | 说明 |
|--------|------|
| `X-RateLimit-Limit` | 该 Key 每分钟最大请求数 |
| `X-RateLimit-Remaining` | 当前窗口剩余可用次数 |
| `X-RateLimit-Reset` | 窗口重置的 Unix 时间戳（秒） |
| `Retry-After` | 建议多少秒后重试（仅 429 时返回） |

#### 配置（application.yml）

```yaml
rate-limit:
  enabled: true
  window-seconds: 60          # 计数窗口（秒）
  default-limit: 20           # 新 Key 的默认配额（可被 t_api_key.rate_limit 覆盖）
```

### 5.6 健康检查 `GET /actuator/health`

**无需 API Key**。返回 AI 服务的连通性状态。

```json
// AI 服务可达
{
  "status": "UP",
  "components": {
    "ai": {
      "status": "UP",
      "details": { "service": "AI analysis service" }
    },
    "...": "..."
  }
}

// AI 服务不可达
{
  "status": "DOWN",
  "components": {
    "ai": {
      "status": "DOWN",
      "details": {
        "service": "AI analysis service",
        "error": "Connection refused"
      }
    },
    "...": "..."
  }
}
```

> 健康检查仅判断 TCP 连接是否可达（不要求 AI 服务路径返回 200），只要连接不拒绝即为 UP。

## 6. 风险评分公式

```
if keywordHit:
    riskLevel = HIGH
    score = 1.0
    reason = "关键词命中"
else:
    score = 0.7 × aiGlitchProb + 0.3 × violenceProb

    score >= 0.6  → HIGH,   reason = "综合评分过高"
    score >= 0.3  → MEDIUM, reason = "综合评分中等"
    score <  0.3  → SAFE,   reason = null (不返回该字段)
```

| 权重 | 说明 |
|---|---|
| `aiGlitchProb` × 0.7 | AI 生成/深度伪造概率，权重最高 |
| `violenceProb` × 0.3 | 暴力内容概率 |

## 7. 错误码汇总

| HTTP 状态码 | code | 触发条件 | 客户端处理建议 |
|---|---|---|---|
| 401 | 401 | 缺少 `X-API-Key` 头或 Key 不匹配 / Key 已吊销或过期 | 重新注册 Key |
| 429 | 429 | 请求频率超过 Key 配额限制 | 等待 `Retry-After` 秒后重试 |
| 400 | 400 | 视频文件超过 10MB | 压缩或分片 |
| 400 | 400 | 缺少必要请求参数 | 检查参数完整性 |
| 400 | 400 | 请求不是 multipart 格式 | 确认 Content-Type |
| 400 | 400 | 业务参数校验失败 | 检查参数合法性 |
| 502 | 502 | AI 分析服务不可达（重试耗尽后） | 稍后重试（建议 ≥30s），可先展示"分析中"占位 |
| 503 | 503 | MySQL 数据库访问异常 | 稍后重试（建议 ≥10s） |
| 500 | 500 | 未预期的服务器内部错误 | 稍后重试，持续出现则上报 |

**401 错误示例：**
```json
{ "code": 401, "message": "Missing or invalid API key" }
```

**429 限流示例：**
```json
{ "code": 429, "message": "请求过于频繁，请稍后再试" }
```

**502 AI 不可用示例：**
```json
{ "code": 502, "message": "AI 分析服务暂时不可用，请稍后重试" }
```

**503 数据库异常示例：**
```json
{ "code": 503, "message": "服务暂时不可用，请稍后重试" }
```

## 9. Redis 键设计

| Key | 类型 | 内容 | TTL |
|---|---|---|---|
| `blacklist:authority` | Set | 权威黑名单 authorId 集合 | 永久 |
| `blacklist:global` | Set | 全局黑名单 authorId 集合 | 永久 |
| `blacklist:temp:{authorId}` | String | 临时拉黑原因文本 | 24h (86400s) |
| `apikey:{keyValue}` | String (JSON) | Key 元数据缓存（id, permissions, status, rateLimit, deviceId） | 5min |
| `ratelimit:{keyValue}:{window}` | String (counter) | 限流窗口请求计数 | 60s |

**黑名单检查顺序**：authority → global → temp（命中任一即停止，返回对应 source）。

**Key 缓存策略**：
- 首次请求时从 MySQL 加载，写入 Redis（TTL 5 分钟）
- Key 被吊销/修改时，主动删除缓存立即生效
- 缓存仅存权限、状态、配额等元数据，不存储 key_value 原文

## 10. AI 服务接口协议

后端以 HTTP multipart 方式调用外部 Python AI 分析服务。

### 请求

```
POST {AI_SERVICE_URL}/v1/analyze/video
Content-Type: multipart/form-data

字段:
  file: <视频文件二进制流>
```

### 期望响应

```json
{
  "aiGlitchProb": 0.70,
  "violenceProb": 0.30,
  "transcription": "视频语音识别结果文本",
  "keywordHit": false
}
```

| 字段 | 类型 | 范围 | 说明 |
|---|---|---|---|
| `aiGlitchProb` | double | 0.0 - 1.0 | AI 生成/伪造内容概率 |
| `violenceProb` | double | 0.0 - 1.0 | 暴力内容概率 |
| `transcription` | String | — | 语音转文字结果 |
| `keywordHit` | boolean | — | 是否命中反诈敏感词库 |

> **调用策略**：连接超时 5s，读取超时 30s。超时或返回 5xx 时，后端自动指数退避重试（最多 3 次，间隔 1s / 2s / 4s）。4xx 客户端错误不重试，直接失败。重试耗尽后返回 502。客户端收到 502 时建议间隔 ≥30s 后重试整个检测请求。

## 12. 客户端接入要点

### 12.1 首次启动 — 注册获取 API Key

1. App 首次启动时，使用 **Root Key**（内置在 App 中，与服务端 `api.key` 一致）调用 `POST /api/v1/keys/register`
2. 请求体：`{ "deviceId": "<设备唯一标识>", "deviceName": "<可选>" }`
3. 服务端返回专属 `apiKey`，App **必须安全持久化存储**（Keychain / Keystore）
4. 后续所有请求使用此专属 Key，不再使用 Root Key

### 12.2 上传重试策略

视频上传可能因网络波动失败，建议 APP 端实现以下重试机制：

| 场景 | 响应码 | 处理 |
|------|--------|------|
| 网络超时（无响应） | — | 重试，最多 3 次，间隔 2s / 4s / 8s |
| AI 服务繁忙 | 502 | 稍后重试，建议间隔 ≥30s |
| 服务端异常 | 500 / 503 | 稍后重试，建议间隔 ≥10s |
| 限流 | 429 | 等待 `Retry-After` 秒后重试 |
| 认证失败 | 401 | 检查 Key，必要时重新注册 |
| 参数/文件错误 | 400 | **不重试**，提示用户检查 |

> **注意**：目前后端不做上传去重。如果上传成功但 APP 没收到响应，重试会导致同一个视频被检测两次、写两条数据库记录。APP 端建议给每次上传生成一个唯一 `requestId`，重试时复用同一个 ID。

### 12.3 上传超时建议

- 服务端连接超时 5s、读取超时 30s，加上 AI 重试最多约 37s（30s + 三次重试的累计等待 7s）
- 建议 APP 端上传超时设置为 **45s**，涵盖服务端所有重试的极端情况

### 12.4 通用注意事项

1. **所有业务请求必须带 `X-API-Key` 头**，否则立即返回 401。
2. **检测接口是同步的**，一个视频从上传到返回结果需要等待 AI 分析完成（含可能的重试），建议客户端设置超时 45s。
3. **视频文件 ≤ 10MB**，超出返回 400。
4. **历史同步推荐用游标模式**：本地存储 `latestId`，下次请求时作为 `afterId` 传入，只拉增量。初次同步传 `page=1` 走分页模式。
5. **`score` 字段可能为 null**（黑名单命中时不返回），客户端展示评分前应判空。
6. **`transcription` 字段可能为 null**（黑名单命中时、SAFE 时没有 reason 等），做好空值保护。
7. **SAFE 结果没有 `reason` 字段**，MEDIUM 和 HIGH 有。
8. **temp 黑名单有 TTL**，过期后自动消失，不需要客户端处理过期逻辑。
9. **id 是单调递增的**，可以用作游标。但不同设备间 id 不连续（全局自增），仅同一设备内有效排序。
10. **健康检查路径 `/actuator/health` 不需要 API Key**，可用于客户端启动时探测后端连通性。
11. **遵守限流**：收到 HTTP 429 时，读取 `Retry-After` 头，等待对应秒数后重试。正常请求时注意 `X-RateLimit-Remaining` 头，提前降速。
12. **Key 被吊销时**，所有请求返回 401。App 应引导用户重新注册（调用 `POST /api/v1/keys/register`）。
13. **服务端 Redis 故障不影响正常使用**：Redis 宕机时认证自动降级到 MySQL，黑名单检查暂跳，限流暂放行。APP 端无需特殊处理。
